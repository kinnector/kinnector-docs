# EDR Policy and Rule Management Guide

This document outlines the workflows and utilities available for defining, baselining, compiling, and deploying security policies (rules, exclusions, sensitive files registry, and allowed CDNs) in the Kinnector EDR suite.

---

## Workflow 1: The Audit-Mode Log Collector (`kinnector-collect`)

To baseline an active host system without risk of breaking operations, deploy in **Audit Mode** to capture potential false positives:

1. **Deploy the Collector**:
   Navigate to [kinnector-collect](file:///home/user/Documents/kinnector/kinnector-collect) and run:
   ```bash
   sudo bash bootstrap_collect.sh
   ```
   This runs the core EDR installer and configures a systemd environment override (`KINNECTOR_AUDIT=true`) so that the agent logs alerts to `/var/log/kinnector/alerts.log` without suspending target processes.

2. **Collect Logs**:
   Allow your target services, pipelines, and workloads to run for 3–7 days.

3. **Baseline Analysis**:
   Inspect `/var/log/kinnector/alerts.log`. Any alerts generated by trusted services must be converted into **Path Exclusions** to prevent blocking in active enforcement mode.

## Workflow 2: Distro-Specific Telemetry Baselines

Standard benchmark parameters and files of interest vary by Linux distribution. We maintain these mappings in [kinnector-protect-community/policies/distro_policies.yaml](file:///home/user/Documents/kinnector/kinnector-protect-community/policies/distro_policies.yaml):

- **Debian / Ubuntu**: Focuses on `/etc/shadow`, `/etc/passwd`, `/etc/pam.d/` permissions, and APT GPG keyrings.
- **RHEL / Rocky / CentOS**: Focuses on certificate stores in `/etc/pki/` and SELinux configuration files.
- **Arch / Manjaro**: Focuses on Pacman keyring configurations (`/etc/pacman.d/gnupg/`).
- **Fedora**: Focuses on system crypto policies (`/etc/crypto-policies/`).

## Workflow 3: Policy-as-Code Compilation Pipeline

Kinnector EDR uses a cryptographically signed binary database `/etc/kinnector/rules.db` containing rule parameters to prevent local tampering.

To modify the policy rules:

1. **Edit JSON Policy Definition**:
   Define rules in [kinnector-protect-community/policies/policy.json](file:///home/user/Documents/kinnector/kinnector-protect-community/policies/policy.json):
   ```json
   {
     "version": 1,
     "exclusions": ["/var/log/", "/sys/"],
     "trusted_clis": [
       { "path": "/usr/bin/ssh", "flags": 16 }
     ],
     "sensitive_files": [
       { "path": "/etc/shadow", "flags": 16 }
     ],
     "allowed_cdns": ["*.ubuntu.com"]
   }
   ```

2. **Compile and Sign**:
   Run the compile helper script inside `kinnector-config`:
   ```bash
   cd kinnector-config
   ./compile.sh
   ```
   - If no private key file is provided, it signs using the development mock key.
   - The script outputs a signed binary `rules.db` with a custom 80-byte security header.

 3. **Deploy**:
   Push the compiled `rules.db` to `/etc/kinnector/rules.db`. The agent's background directory watcher will instantly detect the change, verify the cryptographic signature against the kernel keyring public keys, and hot-reload rules in real time.
